# Maintaining this skill (self-improvement protocol)

← Back to [SKILL.md](../SKILL.md). Treat this skill as a **living asset**, not a
frozen tool. While running any tier, when a restoration surfaces one of the
findings below, fix the skill itself — not just the current output — so the next
run benefits. Then commit (see "Commit discipline" below).

**Three triggers — when you must update the skill:**

1. **An obvious problem.** A script bug, a gate (`quality-gate.ts`, Stage 3 acceptance) that misfires, a wrong/misleading instruction, or a stale file path / flag / script name in the docs.
2. **A process that can be optimized.** A manual step you performed by hand that a script flag could do, a faster stage ordering, or a hand-fix you repeated across chunks that the pipeline should automate.
3. **A new or missed npm package identity.** A vendored chunk whose stripped basename is **not** in `CHUNK_NAME_REGISTRY`, a public vendor file that was about to become a hand-written compatibility body for a stock package, or an external tool/library worth adopting into the skill itself.

**Route each finding to its home:**

| Finding                                                                                         | Where it goes                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| New vendored npm package (e.g. `./react-hook-form-XXXX.js`)                                     | Add the stripped basename to `CHUNK_NAME_REGISTRY` in `scripts/resolve-npm-imports.ts` (`namedOnly` vs `defaultName`); add export names to `ALIAS_REGISTRY` if they appear as cryptic aliases; add public `vendor/<package>.ts[x]` aliases to `PUBLIC_NPM_VENDOR_SHIMS` in `scripts/quality-gate.ts`, exact high-confidence generic provenance chunks (for example `lib-*` / `src-*` package bodies) to `PUBLIC_NPM_VENDOR_SOURCE_CHUNKS`, and a vendor-file API fingerprint to `PUBLIC_NPM_VENDOR_API_FINGERPRINTS` when the package exposes distinctive public symbols that survive renaming (for example React Intl's `useIntl`/`FormattedMessage`); rely on the package-name fallback only as a safety net for `vendor/<package>.ts[x]` files whose names exactly match a declared dependency, not as a substitute for registering generic chunk/provenance/API fingerprints; include fail/pass tests so a hand-written compatibility implementation cannot land and the nearest `package.json` must declare the package root; refresh the `npm-leaf` example list in [boundaries.md](boundaries.md) + the registry mention in [stage-2-restore.md](../stages/stage-2-restore.md). |

**Safety guard — never destabilize the run you're in:**

- Data-only and doc edits (registry entries, caveats, doc fixes) are safe — apply them inline.
- A **behavior-changing script edit must ship with a green test**: add/extend the `*.test.ts` and run `bun test` from the skill dir before committing. Never leave the suite red.
- If a fix is too risky to validate mid-restoration (it would change the in-flight pipeline's behavior), make the smallest safe note instead — flag it with a TODO in the relevant doc or via `spawn_task` — and keep the user's restoration moving. Don't block delivery on a speculative refactor.

**Commit discipline (each turn):**

- At the end of every turn, commit. Keep **skill self-improvements in their own commit**, separate from restoration output.
- Skill commits: scope them `skill(deobfuscate-javascript): <what changed>` (e.g. `add react-hook-form to CHUNK_NAME_REGISTRY`).
- Restoration progress: its own checkpoint commit, following the repo's existing `app-main: …` convention.
- Workspace intermediates under `.deobfuscate-javascript/` stay gitignored (already configured) — never commit those.
